# The metabolic frontier

This directory is not a sales funnel and records no revenue. It is the estate's
interface to the only grader that cannot share its priors: a person or team who
can decline to exchange scarce resources for an outcome. A technical capability
becomes an income hypothesis only after it names all of the following:

1. a buyer and a painful bounded problem;
2. a specific deliverable rather than a capability adjective;
3. local evidence that exists now;
4. an external test that could reject the offer; and
5. a dated kill condition.

`station market` makes those fields executable. `PAID` requires an existing
local receipt pointer. Interest is not payment; a model's confidence is not
interest; a polished packet is not either.

## Why this is a real frontier

Agent adoption is moving from one-off chat toward repeatable workflows, while
current guidance emphasizes guardrails, explicit tool boundaries, testing,
documentation, monitoring, and human control. The estate has working artifacts
for those missing joints: provider-bound sessions, task-risk model routing,
adversarial holdouts, hash-pinned results, append-only state, and a refusal to
turn a failed campaign into a result. That is not a universal business; it is a
narrow but legible capability surface.

Primary grounding:

- [NIST AI RMF](https://www.nist.gov/itl/ai-risk-management-framework) frames
  trustworthiness across design, development, use, and evaluation.
- [NIST AI RMF Playbook](https://airc.nist.gov/airmf-resources/airmf/5-sec-core/)
  calls for documented, repeatable test/evaluation/verification/validation and
  monitoring processes.
- [OpenAI's practical agent guide](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/)
  describes agents as workflow actors operating within clearly defined guardrails.
- [OpenAI's third-party evaluation playbook](https://openai.com/index/trustworthy-third-party-evaluations-foundations/)
  identifies fixed loops and tool interfaces as useful foundations for coding-agent evaluation.

These sources justify investigating a need; they do **not** validate a buyer,
price, conversion rate, or legal/compliance claim.

## Three hypotheses, not three promises

| Thesis | What is exchanged | External falsifier |
|---|---|---|
| Agent assurance sprint | A bounded diagnostic: map one tool-using agent to an action tier, concrete failure cases, evidence routes, and an operator handoff. | No qualified buyer requests a scoped diagnostic by its due date. |
| Evidence-gate implementation | A narrow implementation engagement: install reproducible change/claim gates around one existing agent workflow. | No team requests a technical scope after reviewing the proof packet. |
| Bounded discovery lab | A pre-registered search/evaluation campaign that can return a negative result with receipts. | No research/operations team values a falsifiable result over an unconstrained prototype. |

The sequence is intentional: diagnose before implementation; implement before
research. Do not sell a general “alien architecture.” Sell one bounded
deliverable whose truth conditions survive a cold reader.
